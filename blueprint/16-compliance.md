## Overview

This document defines the legal, compliance, and safety framework governing all content produced, published, and managed by the Faceless Viral OS platform. It is binding for all engineers, operators, and AI agents operating within the system. Violating these rules exposes operators to platform bans, copyright claims, FTC penalties, and AGPL license violations. Every engineer building on this platform must read this document before writing code that touches content generation, asset ingestion, or publishing.

## 1. Originality and Copyright Rules

### 1.1 What Counts as Original Content

Content is considered original and safe to publish when:

| Criterion | Requirement |
|---|---|
| Script | Generated by our AI pipeline from a topic brief; not copied verbatim from any existing source |
| Voiceover | Synthesized using a licensed TTS voice or a cloned voice with explicit consent of the voice owner |
| Visual Assets | Either AI-generated, sourced from approved royalty-free libraries, or created by the operator |
| Music/Audio | Licensed from approved royalty-free sources only (see Section 1.2) |
| Thumbnail | Original composition; no stock photos used without license; no face swaps of public figures |
| Title/Description | Written by AI or operator; not copied from competing channels |

Content that combines licensed components is considered a derivative work and must respect the terms of the most restrictive license in the composition.

---

### 1.2 Music: Royalty-Free Only

**Approved Sources:**

| Source | License Type | Cost | Attribution Required | Notes |
|---|---|---|---|---|
| YouTube Audio Library | YouTube's Audio Library License | Free | Sometimes (track-specific) | Safe only for YouTube; not cleared for TikTok/Instagram |
| Epidemic Sound | Subscription license | ~$15–$49/month | No | Cleared for all major platforms including monetized content |
| Pixabay Music | Pixabay License / CC0 | Free | No | Good for B-roll and backgrounds |
| Artlist | Annual subscription | ~$199/year | No | Cleared for commercial and social media use |
| Musicbed | Per-license or subscription | Varies | No | High quality; better for long-form |
| Bensound | Free tier + Premium | Free/paid | Yes (free tier) | Verify license tier per track |
| ccMixter | Various CC licenses | Free | Depends on license | Must check per-track — avoid CC-BY-NC for commercial use |

**Strictly Prohibited Music Sources:**
- Any commercially released music not explicitly licensed (Spotify/Apple Music tracks, etc.)
- Music ripped from other YouTube videos
- Music described as "no copyright" in YouTube video titles (legally meaningless)
- AI-generated music that doesn't come with a clear commercial license

**Music Enforcement Rules:**
- Every video in the system must have a `music_asset_id` foreign key to the asset provenance table
- Background music volume must be normalized to -20 dB or lower under voiceover
- The platform must check music source against the approved sources list before allowing a render to proceed
- YouTube Content ID can flag tracks even from "royalty-free" libraries if the library has registered the track — maintain a blocklist of known Content ID-registered tracks

---

### 1.3 Stock Footage Provenance Requirements

All stock video and image assets used in production must have the following fields populated in the `asset_provenance` table before a video render can begin:

| Field | Required | Notes |
|---|---|---|
| `source_url` | Yes | Original URL where the asset was found/downloaded |
| `provider` | Yes | `pexels`, `pixabay`, `unsplash`, `storyblocks`, `runway`, `self`, `operator_upload` |
| `provider_asset_id` | Yes | Provider's internal ID for the asset |
| `license_type` | Yes | e.g., `pexels_license`, `pixabay_license`, `cc0`, `cc_by`, `custom` |
| `license_url` | Yes | URL to the full license text |
| `license_expiry` | No | Required for subscription-based licenses (Storyblocks, Epidemic Sound) |
| `usage_rights` | Yes | `commercial`, `editorial_only`, `restricted` |
| `attribution_required` | Yes | Boolean; if true, attribution text must appear in video description |
| `attribution_text` | Conditional | Required if `attribution_required = true` |
| `downloaded_at` | Yes | Timestamp of download |
| `cached_r2_key` | Yes | R2 path of our local copy |
| `checksum_sha256` | Yes | SHA-256 of downloaded file (for integrity verification) |

**Pre-render Gate:** The rendering pipeline must call `AssetProvenanceService.validate(assetIds[])` before beginning any render job. This function:
1. Verifies all assets exist in the table
2. Checks no `license_expiry` is in the past
3. Confirms all `usage_rights` are `commercial`
4. Returns a signed "provenance check passed" record attached to the render job

### 1.4 AI-Generated Content Disclosure

Platform policy on disclosure of AI-generated content:

| Platform | Current Policy | Our Requirement |
|---|---|---|
| YouTube | Requires disclosure label for "realistic-looking" AI content in sensitive categories (politics, health, news) | Flag all AI-generated voiceovers + visuals; use YouTube's built-in disclosure toggle in video settings |
| TikTok | AIGC (AI Generated Content) label required for all AI-generated realistic content | Automatically add AIGC label via API when content is AI-generated |
| Instagram | Meta requires disclosure for AI-generated content in ads; organic policy evolving | Add "AI-generated content" note in description for all AI-produced videos |

**System Behavior:**
- Every video object in the DB has an `ai_generated_components` JSONB field tracking which components (script, voiceover, visuals, music) were AI-generated
- At publish time, the publisher module reads this field and applies the appropriate disclosure flags per platform
- Operators cannot disable disclosure flags on platforms that mandate them

### 1.5 Script Originality Rules

| Rule | Description |
|---|---|
| No verbatim copying | Scripts must not reproduce more than 15 consecutive words from any identifiable source without transformation |
| Transformation doctrine | Using factual information (news, statistics, public records) is permitted; the expression must be our own |
| No SEO scraping and republishing | Do not fetch competitor video transcripts and republish them with light rewording |
| Citation of facts | Statistics and data points should reference their source in the video description |
| Script similarity check | Before accepting a generated script, run a cosine similarity check against the last 50 scripts for the same channel (threshold: < 0.85 similarity) to prevent repetition |

## 2. Inspiration vs Cloning Boundaries

This is the most important section for the Competitor Intelligence module. The line between legitimate competitive intelligence and content cloning is where platforms ban accounts and creators file copyright claims.

### 2.1 What Is Allowed

| Activity | Description | Implementation Notes |
|---|---|---|
| Format reverse engineering | Studying hook structure, pacing, section flow, intro/outro patterns | Extract metadata from transcripts; do not store raw transcripts |
| Hook structure analysis | Identifying what makes a successful hook (question hooks, stat hooks, controversy hooks) | Store pattern classifications, not copied text |
| Pacing study | Measuring average segment length, cut frequency, information density | Analyze timing data from transcript timestamps |
| Niche validation | Confirming a niche is viable by measuring competitor view counts, growth rate, engagement | Aggregate statistics only; no PII |
| Topic clustering | Identifying what topics perform well in a niche | Cluster by theme, not by copying titles |
| Thumbnail style analysis | Understanding color palettes, text-to-image ratio, emotion portrayal | Visual metadata analysis; no image cloning |
| Engagement pattern analysis | Understanding comment sentiment, what questions viewers ask | Aggregate sentiment; no individual comment scraping beyond API limits |

### 2.2 What Is NOT Allowed

| Activity | Why Prohibited | Technical Prevention |
|---|---|---|
| Script copying | Copyright infringement; plagiarism | Script similarity check blocks scripts with > 0.85 similarity to known sources |
| Thumbnail tracing | Copyright infringement if traced from original creative work | No image-to-image pipeline that takes competitor thumbnails as input |
| Channel name mimicry | Trademark issues; brand confusion; platform ban risk | Operator checklist during channel creation |
| Fake engagement | Platform TOS violation; account ban; potential FTC violation | No engagement-buying integrations permitted |
| Sub4sub schemes | Platform TOS violation | No integrations with third-party engagement services |
| Purchasing views/followers | Platform TOS violation; FTC deceptive practices | Explicitly blocked in platform integrations |
| Republishing competitor content | Copyright infringement | Pipeline does not accept competitor video files as input |
| Scraping beyond API limits | Platform TOS violation | All data collection routes through official APIs only |

### 2.3 Competitor Intelligence Module — UI Disclosure

The Competitor Intelligence dashboard must display the following banner permanently visible during use:

> "This tool provides aggregate performance insights for educational and competitive analysis purposes. Copying scripts, cloning thumbnails, or reproducing any content creator's original expression is a violation of copyright law and platform Terms of Service. Faceless Viral OS does not permit or facilitate content cloning."

Additionally:
- All competitor data views must show only aggregate metrics, not individual creator's raw content
- Transcript extraction from competitor videos is not available as a feature — only structural metadata (pacing, segment count, word count)
- Any feature request that would enable scraping or copying competitor content must be escalated to legal review

## 3. Asset Provenance System

### 3.1 Database Schema

```sql
CREATE TABLE asset_provenance (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id    UUID NOT NULL REFERENCES workspaces(id),
  provider        TEXT NOT NULL,
  provider_asset_id TEXT,
  source_url      TEXT NOT NULL,
  license_type    TEXT NOT NULL,
  license_url     TEXT,
  license_expiry  TIMESTAMPTZ,
  usage_rights    TEXT NOT NULL CHECK (usage_rights IN ('commercial', 'editorial_only', 'restricted')),
  attribution_required BOOLEAN NOT NULL DEFAULT false,
  attribution_text TEXT,
  asset_type      TEXT NOT NULL CHECK (asset_type IN ('video', 'image', 'audio', 'music', 'font')),
  cached_r2_key   TEXT,
  checksum_sha256 TEXT NOT NULL,
  file_size_bytes BIGINT,
  downloaded_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE video_assets (
  video_id    UUID NOT NULL REFERENCES videos(id),
  asset_id    UUID NOT NULL REFERENCES asset_provenance(id),
  usage_context TEXT, -- 'background', 'broll', 'music', 'thumbnail', 'intro'
  PRIMARY KEY (video_id, asset_id)
);
```

### 3.2 Automated License Check Before Publish

Implemented as a BullMQ job step in the publish pipeline:

```
PublishJob
  Step 1: ProvenanceCheckStep
    - Load all video_assets for this video
    - For each asset: check license_expiry, usage_rights = 'commercial'
    - If any check fails: abort job, set video status = 'publish_blocked', notify operator
  Step 2: (only if Step 1 passes) PlatformPublishStep
```

### 3.3 Audit Trail

Every asset download and usage is logged in the `asset_audit_log` table:
- `action`: `downloaded`, `used_in_render`, `license_check_passed`, `license_check_failed`
- `actor`: `system` (automated) or user ID (operator upload)
- `timestamp`: exact time
- `metadata`: JSONB with relevant context

## 4. Content Moderation

### 4.1 Prohibited Content Categories

The following content types are absolutely prohibited from the automated pipeline:

| Category | Examples | Prevention |
|---|---|---|
| Adult/NSFW | Sexual content, nudity, explicit material | LLM-based script review + keyword blocklist |
| Violence | Graphic violence, gore, threats | LLM-based script review + keyword blocklist |
| Hate speech | Content targeting protected groups | LLM-based review + classifier |
| Misinformation | Health misinformation, election interference, COVID denial | Fact-check flag on health/political topics |
| Dangerous activities | How-to content for illegal activities | Topic-level blocklist |
| Spam / deceptive content | Misleading titles, fake testimonials | Operator checklist |
| Private information | Doxxing, PII exposure | PII scanner on all scripts |

### 4.2 Script Safety Check (Pre-Render Gate)

Before any script proceeds to TTS/render, it must pass the **Script Safety Review**:

```
ScriptSafetyReview(script: string): SafetyResult
  1. Keyword blocklist scan (fast, sync)
  2. LLM content safety classifier (async)
     - Model: Claude Haiku or GPT-4o-mini (cost-optimized)
     - System prompt: "You are a content safety reviewer. Classify this script 
       for: NSFW, violence, hate speech, misinformation, dangerous advice, spam. 
       Return JSON: {safe: boolean, flags: string[], confidence: number}"
  3. PII detection (regex-based: emails, phone numbers, addresses, SSNs)
  4. If any flag raised: hold for operator review; do not auto-publish
  5. Log result to script_safety_reviews table
```

**Thresholds:**
- If `confidence > 0.85` and `safe = false`: auto-reject, notify operator
- If `confidence 0.60–0.85` and `safe = false`: hold for manual review
- If `confidence < 0.60`: pass but flag for operator spot-check

### 4.3 Platform Community Guidelines Compliance Checklist

| Platform | Key Rules to Enforce |
|---|---|
| YouTube | No misleading metadata; no spam; no artificial engagement; no copyright-infringing content; age-gating for mature content |
| TikTok | No dangerous challenges; no misinformation; no adult content; no paid political ads without disclosure |
| Instagram | No nudity; no hate speech; no false information; no spam; no intellectual property violations |

A platform-specific checklist is presented to operators during the pre-publish review screen. Operators must acknowledge the checklist before manual publish. Automated publish skips the UI checklist but the system-level safety checks are non-bypassable.

## 5. Affiliate and Sponsorship Disclosure

### 5.1 FTC Requirements

The FTC requires clear and conspicuous disclosure of material connections to brands, including:
- Affiliate links (earning commission on sales)
- Sponsored content (paid to create content about a product)
- Free products received in exchange for coverage
- Personal financial interest in a company mentioned

**"Clear and conspicuous"** means:
- Placed prominently at the beginning of descriptions (not buried at the bottom)
- Not hidden behind a "more" expand button
- Written in plain language (not just "#ad" buried in hashtags — though "#ad" is acceptable when prominent)
- In the same language as the content

### 5.2 System Implementation

When a video's `monetization_links` field is non-empty (contains affiliate or sponsored links):

1. The system automatically prepends the following to the video description:
   ```
   DISCLOSURE: This video contains affiliate links. I may earn a commission on 
   qualifying purchases at no additional cost to you.
   ```
   (Or sponsor-specific language if `sponsorship_disclosure_text` is set on the video)

2. The disclosure is non-removable via the UI once affiliate links are attached — operators cannot publish a video with affiliate links without the disclosure

3. For sponsored content, operators must explicitly mark `content_type = 'sponsored'` which triggers YouTube's paid promotion flag (enabled via API) and the platform-specific disclosure mechanisms

4. All disclosure text and timestamps are logged in the `disclosure_audit_log` table

### 5.3 Per-Platform Disclosure Requirements

| Platform | Mechanism |
|---|---|
| YouTube | Use `paidProductPlacementDetails.hasPaidProductPlacement = true` in API + verbal/visual disclosure |
| TikTok | Use branded content toggle in API (`branded_content_type: 'branded'`) + description disclosure |
| Instagram | Use branded content tag feature + description disclosure |

## 6. Rate Limiting and Anti-Spam

### 6.1 API Rate Limits (Per Provider)

See Document 15 (Integrations) for per-provider rate limits. All provider API calls are routed through the `ProviderRateLimiter` service, which:
- Tracks requests per rolling minute/hour/day window using Redis counters
- Applies backpressure to the BullMQ queue before limits are hit (at 80% of limit)
- Implements exponential backoff on 429 responses

### 6.2 Publishing Rate Limits (Platform-Specific)

| Platform | Maximum Posts | Minimum Interval | Notes |
|---|---|---|---|
| YouTube | 6 videos/day/channel (quota-constrained) | 4 hours between uploads | System enforces 1 upload/4 hours/channel |
| TikTok | 100 posts/day (API limit) | 30 minutes | System enforces 1 post/hour for safety |
| Instagram | 25 posts/24 hours | 1 hour | Enforce at API level |

**System-enforced limits (non-bypassable):**
- Maximum 4 video uploads per channel per day (YouTube, even if quota allows more)
- Minimum 2 hours between any two posts to the same channel on any platform
- Maximum 10 scheduled posts in queue per channel at any time

### 6.3 Anti-Spam Measures

- **Duplicate content detection:** Before scheduling a video for publish, compute a content fingerprint (title embedding + script embedding) and compare against the last 30 published videos on the same channel. Block if similarity > 0.90.
- **Velocity throttling:** If more than 3 videos are published within 12 hours on a single channel, require explicit operator confirmation before the 4th.
- **Channel warm-up protocol:** New channels (< 30 days old, < 1,000 subscribers) are limited to 1 post per day per channel until manual override by operator.

### 7.2 Log Retention Policy

| Phase | Retention Period | Storage |
|---|---|---|
| Phase 1 (private tool) | 90 days | PostgreSQL + R2 archive |
| Phase 2 (public SaaS) | 2 years | PostgreSQL (hot: 90 days) + R2 archive (cold: 2 years) |

**Archival process (Phase 2):**
- At 90 days, batch job exports logs to Parquet files in R2 cold storage
- Logs older than 90 days are purged from PostgreSQL
- Archives are queryable via DuckDB for incident investigation

### 7.3 Audit Log Immutability

- Audit log records are insert-only (no updates, no deletes)
- Application roles do not have DELETE privilege on audit tables
- In Phase 2, consider append-only ledger storage (Amazon QLDB or Immudb) for financial audit trails

## 8. AGPL-3.0 Boundary Strategy: MoneyPrinterV2

### 8.1 What MoneyPrinterV2 Is

MoneyPrinterV2 (github.com/FujiwaraChoki/MoneyPrinterV2) is an open-source Python tool licensed under the AGPL-3.0 license. It automates the creation of faceless YouTube Shorts by combining AI script generation, TTS, stock video assembly, and automated uploads. It serves as a reference for the product space we are operating in, demonstrating that automated faceless content production is technically viable.

It is NOT an SDK, library, or framework we are building on top of. It is a reference point only.

### 8.2 AGPL-3.0 — What It Means

The GNU Affero General Public License v3.0 is a strong copyleft license with a network use clause:
- Any software that incorporates AGPL-3.0 code (including modified versions) must release all source code under AGPL-3.0
- The network use clause extends this to **software offered as a service** — if you run AGPL-3.0 code on a server and users interact with it over a network, you must provide the source code to those users
- There is no "SaaS loophole" with AGPL as there is with GPL

**If we incorporate any MoneyPrinterV2 source code:**
- Faceless Viral OS's entire codebase would become subject to AGPL-3.0
- We would be required to publish all source code publicly
- Our proprietary SaaS business model would be legally untenable

**This is a hard line. Zero tolerance.**

### 8.3 What We Are Allowed to Do

| Activity | Permitted | Notes |
|---|---|---|
| Read the MoneyPrinterV2 source code | Yes | Reading code to understand architecture is permitted |
| Learn from its architecture | Yes | Architectural ideas are not copyrightable |
| Use it as a feature reference document | Yes | "MoneyPrinterV2 does X; we should also do X" is fine |
| Understand what APIs it uses | Yes | API choices are not copyrightable |
| Understand its workflow sequencing | Yes | Workflows/processes are not copyrightable (though code expressing them is) |
| Run it for personal evaluation | Yes | Using open source software is permitted |

### 8.4 What We Must NOT Do

| Activity | Prohibited | Risk |
|---|---|---|
| Copy any source file, function, or class | Yes | AGPL infection of our entire codebase |
| Copy variable names, comment blocks, or string literals | Yes | Potential copyright infringement |
| Use it as a library dependency (npm/pip import) | Yes | AGPL infection |
| Fork it and add features without AGPL compliance | Yes | License violation |
| Copy its prompt templates verbatim | Yes | Copyright (prompts are creative expression) |
| Use its Docker configuration | Yes | Files are copyrighted |
| Copy its database schema designs | Yes | Potential copyright; also creates similarity risk |

### 8.5 Clean-Room Implementation Protocol

All features that were inspired by MoneyPrinterV2 must follow this protocol:

**Step 1: Document Extraction (Permitted)**
> Create a feature specification document that describes WHAT the feature does (in our own words), NOT HOW the MoneyPrinterV2 code implements it. This document lives in `docs/feature-specs/` and is the only permitted artifact derived from studying MoneyPrinterV2.

**Step 2: Implementation Independence (Required)**
> The engineer implementing the feature must NOT have the MoneyPrinterV2 source code open while writing our code. Implementation begins from the feature specification only.

**Step 3: Independent Review (Recommended)**
> Have a second engineer review the implementation who has NOT studied the MoneyPrinterV2 code for that feature.

**Step 4: Documentation**
> In the PR description, explicitly state: "This feature was implemented independently based on the feature spec in `docs/feature-specs/[feature].md`. No MoneyPrinterV2 source code was referenced during implementation."

### 8.6 Features Inspired By MoneyPrinterV2 (Feature Reference Only)

The following features are in our roadmap because MoneyPrinterV2 demonstrated them. The implementations are entirely original:

| Feature | MoneyPrinterV2 Concept | Our Implementation Approach |
|---|---|---|
| Script generation | Uses LLM to generate video script | Model Router with multi-provider support and cost optimization |
| TTS pipeline | Google TTS + ElevenLabs | Provider-agnostic TTS adapter with quality/cost tiers |
| Stock video assembly | Pexels API + MoviePy | Pexels + Pixabay + Runway via FFmpeg-based pipeline |
| Subtitle generation | Whisper transcription + rendering | Whisper API → ASS format → FFmpeg subtitle filter |
| Auto-upload | YouTube API upload | Multi-platform publisher with resumable upload support |
| Niche research | YouTube search-based | Combined YouTube + TikTok research with trend scoring |

### 8.7 Recommended Stance

**Build entirely from scratch.** The Faceless Viral OS architecture is substantially more sophisticated than MoneyPrinterV2 (multi-platform, multi-tenant, cost governance, model routing, analytics feedback loop, etc.). The overlap in features does not require reading MoneyPrinterV2's implementation. Treat it as a confirmed proof-of-concept that the problem space is viable, and build our solution independently from the TypeScript monorepo foundations up.

If any team member is uncertain whether a piece of code is too similar to MoneyPrinterV2, the answer is: **rewrite it from scratch, no discussion needed.** The legal risk of AGPL contamination far outweighs the cost of a rewrite.

## 9. Platform Terms of Service Compliance

### 9.1 YouTube TOS Key Points

- Automated uploads are permitted via the YouTube Data API with proper OAuth
- Mass uploading or spam is prohibited
- Cannot use the API to circumvent platform policies
- Content must comply with Community Guidelines at time of upload
- Cannot programmatically manipulate view counts, likes, or subscriptions

### 9.2 TikTok TOS Key Points

- Content Posting API requires business account approval
- Cannot post misleading content or impersonate others
- No automation of engagement (likes, follows, views)
- Content must be appropriate for the platform's 13+ age policy

### 9.3 Meta/Instagram TOS Key Points

- Must comply with Instagram Community Guidelines
- Graph API usage subject to Meta's Platform Terms
- Cannot store user data beyond what is needed for the declared purpose
- Automated posting must use official Graph API (no scraping, no browser automation)

### 9.4 Compliance Review Cadence

| Review Type | Frequency | Owner |
|---|---|---|
| Platform TOS review | Quarterly | Platform Integration Lead |
| API usage compliance check | Monthly | DevOps |
| Content policy review | Quarterly | Operator |
| AGPL boundary review | On any new external library addition | Engineering Lead |
| FTC disclosure review | Annually or when FTC guidance changes | Operator/Legal |

## Summary: Non-Negotiable Rules

1. Never copy MoneyPrinterV2 source code — zero tolerance
2. Never use music without a commercial license
3. Never publish without asset provenance verified
4. Never skip the script safety check
5. Never publish affiliate content without FTC-compliant disclosure
6. Never scrape competitor content beyond official API limits
7. Never buy engagement of any kind
8. All AI-generated content must be appropriately disclosed per platform requirements
9. Audit logs are immutable and must be retained per the retention policy
10. Platform TOS must be reviewed quarterly — APIs and policies change
